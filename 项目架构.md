# Sync Copy — 项目架构与功能说明

> 本文档是一份可"逆向生成"的完整规格：把它连同 Tauri 2 + Svelte 的基本范式交给 Claude（或任何有足够能力的 LLM），理论上能产出一个行为等价的程序。所以文档中包含产品目标、功能清单、协议细节、状态机、UI 规格、目录结构、依赖清单等所有重建所需的信息。

---

## 1. 产品定位

**Sync Copy** 是一个运行在局域网内的多设备剪切板/文件同步小工具。

- **跨平台**：macOS + Windows（Linux 理论可行但未验证）
- **形态**：一直悬浮在所有窗口最前的小浮窗（320×420），可收缩为 48×48 的悬浮球
- **核心价值**：在一台设备上复制的内容自动同步到其它设备，做到"像在同一台电脑上复制-粘贴"
- **不依赖外部服务**：无中心服务器，所有通信走局域网 HTTP；完全对等（P2P）
- **一次配置多机共用**：一人多机的场景优化（家里 iMac + 公司 MacBook + 测试 Windows）

## 2. 核心功能

### 2.1 剪切板同步
- **文本**：每秒轮询系统剪切板，变化即广播到所有小组成员，对方收到后自动写入自己的剪切板，可以直接 `Cmd/Ctrl+V` 粘贴
- **图片**：同样机制，PNG 编码后加密传输，接收端用 `arboard::set_image` 写入系统剪切板；支持在 Word / 微信 / Keynote / 备忘录等应用直接粘出图片
- **去重**：按内容 SHA-256 hash 识别重复项，5 秒内相同内容不重复广播
- **环路防止**：从远端写入本地剪切板时记录 hash，下一次轮询看到同 hash 不再回传

### 2.2 文件传输
- **触发**：把文件拖到浮窗上松手
- **上限**：单文件 5 MB（超过拒绝发送并红色横幅提示）
- **确认**：接收端弹确认框 `📎 收到 xxx.pdf (3.2 MB) 来自 <设备名> / [拒绝] [保存]`
- **保存位置**：系统 Downloads；同名文件自动加 `_1` / `_2` 后缀
- **点击文件历史条目**：在 Finder/Explorer 里定位到实际保存位置
- **三种状态**：`已发送`（发送端）、`已保存`（接收端成功）、`保存失败：<原因>`（磁盘/权限问题）

### 2.3 历史记录
- 浮窗中部纵向列表，最多 50 条，自动按时间倒序
- 文本：文字预览 + 单击复制回系统剪切板 + 变绿闪烁 + `已复制` 微提示
- 图片：显示缩略图 + `宽×高`；单击复制回系统剪切板
- 文件：显示 📎 + 文件名 + 大小 + 状态；单击在文件管理器里定位
- 每条右侧常驻灰色 `✕` 按钮（悬停变红），点击删除**并同步删除其它设备上的同一条**
- 同步删除用 content_hash 跨机器识别"同一条"

### 2.4 小组管理

#### 2.4.1 加入
- 点浮窗顶部蓝色 `加入` 胶囊 → 弹「加入小组」面板
- 输入框填对方设备的 `IP:PORT`（在对方浮窗**左下角**显示，点击可复制）
- 点击「加入」后：
  - 本机启动 HTTP 服务（若还没启动）
  - 向目标 POST `/handshake`
  - 等待对方审批，成功后完成密钥协商，连接建立
- 密码不再是概念的一部分，身份由审批弹框的人工确认决定

#### 2.4.2 审批（分布式）
- 小组内**任何设备**收到新设备的握手 → **所有在线设备**都弹同一个审批框
- **谁先按谁作数**：任一设备点「同意」或「拒绝」，其它设备的弹框自动消失，决定全组生效
- 弹框内显示申请方设备名 + 30 秒倒计时（`还剩 N 秒未响应视为拒绝`）
- 30 秒内无人响应 → 统一关闭弹框，申请方收到 408

#### 2.4.3 信任/封禁传播（Gossip）
- 任一成员同意了设备 X → 向所有其它成员广播 `/peers/trust` → 其它成员把 X 存入 `approved_device_ids`
- 任一成员拒绝了设备 X → 广播 `/peers/ban` → 其它成员把 X 存入 `banned_device_ids`，若当时连接中则直接踢出
- 后续 X 主动连任何成员：
  - 命中 `banned_device_ids` → 直接 403，不弹框
  - 命中 `approved_device_ids` → 跳过弹框直接通过（仍做密钥协商）
  - 都没命中 → 走正常审批流程
- 信任和封禁**互斥覆盖**：信任消息清除同 device_id 的 ban 记录，反之亦然
- 这两个集合**仅存内存**，进程重启清空（设计选择：本地关机后不应长期记忆，重启即"重置信任"）

#### 2.4.4 下线
- 点设置里红色「退出应用」按钮 → 或 `leave_group` 命令
- 流程：先 **broadcast_leave** 告知所有 peer（最多等 1.5 秒），再清空本机状态 + 停 HTTP 服务
- 其它成员收到 `/peers/leave` → 把本机从 `peers` 表移除 + 删除对应 AES 密钥 + 更新状态

#### 2.4.5 心跳（被动掉线检测）
- 后台每 10 秒给所有 peer `GET /ping`（2 秒超时）
- 连续 2 次失败 → 该 peer 被从本地 `peers` 表移除 + 发 `status-updated` 事件
- 单次失败不扣分，避免临时网络抖动误伤
- 上面 2.4.4 的主动 `leave` 走不通时由此兜底

### 2.5 端到端加密

#### 2.5.1 密钥协商
- 每次握手双方各自生成临时 X25519 密钥对（**ephemeral**，仅本次会话有效）
- 交换公钥后做 Diffie-Hellman 得出 32 字节共享秘密
- HKDF-SHA256 扩展为 AES-256 会话密钥
- salt = `b"sync-copy-v1-salt"`，info = `b"sync-copy-v1:aes-256-gcm"`
- **每一对 peer 独立一把密钥**，互不共用
- 密钥**只存内存**（`Arc<RwLock<HashMap<device_id, [u8;32]>>>`）；进程重启即重新协商

#### 2.5.2 报文加密
- 所有剪切板/文件 payload 走 AES-256-GCM
- 每条消息独立 12 字节随机 nonce（`OsRng`）
- 密文 + nonce 都 base64 编码后放进 JSON
- 握手请求本身不加密（只有 X25519 公钥，无机密信息）；身份校验由接收方的审批弹框承担

## 4. 系统架构

### 4.1 模块划分

**Rust 后端**（`src-tauri/src/`）：

| 模块 | 职责 |
|---|---|
| `main.rs` | 入口，调用 `lib::run()` |
| `lib.rs` | Tauri Builder；注册命令；启动托盘、剪切板线程、心跳任务 |
| `config.rs` | `Config` 持久化（`device_id`, `device_name`, `port`, `peer_hint`），存到 `ProjectDirs` |
| `state.rs` | `AppState` 聚合所有共享状态；`PeerKeys`, `ForwardedApprovalInfo`, `PendingFileSave` 等值类型 |
| `commands.rs` | 所有 `#[tauri::command]`，是前端 IPC 的唯一入口 |
| `clipboard.rs` | 独立 std thread 持有 `arboard::Clipboard`，1s 轮询 + mpsc channel 接收 `SetTextSuppress` / `SetImageSuppress` |
| `crypto.rs` | X25519 临时密钥对、HKDF 派生、AES-256-GCM encrypt/decrypt |
| `history.rs` | `History` 环形队列（`VecDeque`，上限 50），按 content_hash 去重/删除 |
| `peer.rs` | `PeerRegistry`：`HashMap<device_id, Peer>`，线程安全 |
| `network/mod.rs` | 网络模块入口 |
| `network/protocol.rs` | 所有请求/响应 DTO（serde Serialize/Deserialize） |
| `network/server.rs` | axum Router + 所有端点 handler；启动/关停通过 oneshot 通知 |
| `network/client.rs` | reqwest 客户端；每类广播 + 握手调用 |
| `network/health.rs` | 心跳检测后台任务 |

**前端**（`src/routes/+page.svelte` 单文件 SPA）：
- 没有路由拆分（小工具，一屏搞定）
- 使用 SvelteKit 的 adapter-static + `ssr = false`，产物为纯静态文件
- 状态用 Svelte 5 runes：`$state` / `$derived` / `$effect`
- Tauri IPC 用 `@tauri-apps/api/core` 的 `invoke`
- Tauri 事件用 `@tauri-apps/api/event` 的 `listen`

### 4.2 AppState 结构（核心）

```rust
pub struct AppState {
    pub config: RwLock<Config>,
    pub status: RwLock<ConnectionStatus>,
    pub history: History,
    pub peers: PeerRegistry,

    // 剪切板线程通道
    pub clipboard_tx: Mutex<Option<mpsc::Sender<ClipboardCmd>>>,

    // HTTP server 关停信号
    pub server_shutdown: Mutex<Option<oneshot::Sender<()>>>,

    // 出站消息单调递增 seq
    pub seq: AtomicU64,

    // 接收去重：每个 origin_device_id 最后见过的最大 seq
    pub last_seen_seq: RwLock<HashMap<String, u64>>,

    // 握手审批
    pub pending_approvals: Mutex<HashMap<String, oneshot::Sender<bool>>>,
    pub forwarded_approvals: Mutex<HashMap<String, ForwardedApprovalInfo>>,

    // 文件保存审批
    pub pending_file_saves: Mutex<HashMap<String, PendingFileSave>>,

    // 端到端加密密钥（每 peer 一把）
    pub peer_keys: PeerKeys,

    // 信任/封禁传播列表（内存态，重启清空）
    pub approved_device_ids: RwLock<HashSet<String>>,
    pub banned_device_ids: RwLock<HashSet<String>>,
}
```

### 4.3 连接状态机

```
Idle ──── join_group(target) ──────┐
  │                                │
  ├── join_group(empty) ─► Listening
  │                                │
Listening ◄─── Error ◄──── Connecting (握手中)
  │                                │
  │             Connected ◄────────┘（握手成功）
  │                ▲
  │                │ 有 peer 加入/退出
  └────────────────┘
```

### 4.4 数据流（复制文本为例）

```
 [用户在 Mac 上 Cmd+C]
        │
 arboard 线程 1s 轮询 → 发现变化
        │
 算 SHA-256 → push_text() → history append
        │
 spawn broadcast_text(state, text)
        │
 对每个 peer：
   取 aes_key[peer.device_id]
   AES-GCM encrypt(text.as_bytes(), random_nonce)
   POST http://<peer>/clipboard {
     origin_device_id, origin_device_name, seq,
     nonce(b64), ciphertext(b64),
     kind: "text"
   }
        │
 [Windows 上 axum 收到 /clipboard]
        │
 验证发送方是已认证 peer
 seen_seq_and_update 去重
 取 aes_key[origin_device_id] → AES-GCM decrypt
        │
 push_text(decrypted, Source::Remote{device_name})
 emit "history-updated" 事件 → 前端刷新
 send ClipboardCmd::SetTextSuppress(text) → 剪切板线程
        │
 arboard 写入系统剪切板 + 更新 last_seen 防回环
        │
 [用户在 Windows 上 Ctrl+V 粘出来]
```

## 5. 网络协议

所有请求都是 JSON body，`Content-Type: application/json`。端口默认 5858。

| 端点 | 方法 | 用途 | 是否需要认证 |
|---|---|---|---|
| `/ping` | GET | 心跳 | 否 |
| `/handshake` | POST | 首次建立连接 / 再握手换密钥 | 走审批或预授权 |
| `/clipboard` | POST | 剪切板广播 | 加密 + 发送方必须是已知 peer |
| `/file` | POST | 文件传输 | 同上 + 用户审批 |
| `/delete_history` | POST | 单条删除同步 | 发送方必须是已知 peer |
| `/history/clear` | POST | 清空历史同步 | 发送方必须是已知 peer |
| `/peers/leave` | POST | 离线通知 | 同上 |
| `/peers/trust` | POST | 信任传播 | 同上 |
| `/peers/ban` | POST | 封禁传播 | 同上 |
| `/peers/approval/forward` | POST | A 转发审批请求给 B | 同上 |
| `/peers/approval/decide` | POST | B 把决定送回 A | 同上 |
| `/peers/approval/dismiss` | POST | A 告诉所有人关掉弹框 | 同上 |

### 5.1 握手

**Req** `POST /handshake`:
```json
{
  "device_id": "uuid",
  "device_name": "Tao's MacBook",
  "listen_port": 5858,
  "pubkey": "base64(x25519 public 32 bytes)"
}
```

**Resp** `200 OK`:
```json
{
  "device_id": "uuid",
  "device_name": "Windows-Desk",
  "peers": [
    { "device_id": "...", "device_name": "...", "addr": "192.168.1.10:5858" }
  ],
  "pubkey": "base64"
}
```

**错误码**：
- `400` bad request / 公钥无效
- `403` 被拒绝 或 不在白名单且审批拒绝
- `408` 审批超时
- `409` device_id 冲突（同一配置被复制到两台机器）

服务端收到握手后：
1. 解析公钥
2. 生成自己的临时密钥对
3. 黑名单检查 → 否则 → 已知 peer 检查（直接 re-key）→ 否则 → 白名单检查（跳过弹框通过）→ 否则 → 进审批流程
4. 审批流程：插入 `pending_approvals[request_id] = tx`，emit 本地弹框事件，broadcast forward 给其它 peer
5. `tx.recv()` 等待决定（30s 超时）
6. 决定确定后：
   - 无论结果都 broadcast dismiss，清理 pending
   - 通过 → 派生 AES key、加入 peers、加入 approved_device_ids、await broadcast_trust（最多 2s）
   - 拒绝 → 加入 banned_device_ids、spawn broadcast_ban
7. 返回响应（含自己的公钥 + peers 列表用于 gossip mesh）

客户端拿到响应后：
1. 用自己的 secret 和对方的 pubkey 派生 AES key，存到 peer_keys
2. 对 resp.peers 里每个未知 peer spawn 一次握手（gossip mesh 扩展）

### 5.4 信任 / 封禁 / 审批转发 / 审批决定 / 审批关闭

结构相似，都包含 `origin_device_id + seq`，以及各自的 payload：

```rust
TrustReq { ..., subject_device_id, subject_device_name }  // /peers/trust 和 /peers/ban
ApprovalForwardReq { ..., request_id, subject_device_id, subject_device_name }
ApprovalDecisionReq { ..., request_id, accept: bool }
ApprovalDismissReq { ..., request_id }
```

### 5.5 Leave / Clear-all / 删除单条

```rust
GroupActionReq { origin_device_id, seq }  // 用于 /peers/leave 和 /history/clear
DeleteHistoryReq { origin_device_id, seq, content_hash }
```

### 5.6 服务端通用校验流程

每个非握手端点的 handler 统一逻辑：
1. 检查 `origin_device_id` 是否在 `peers` 表 → 否则 403 Forbidden
2. 调 `state.seen_seq_and_update(origin_device_id, seq)` 去重 → 重复则 200 OK 静默丢弃
3. 根据 kind 分发到具体业务

### 5.7 ConnectInfo 技巧
- axum `Router.into_make_service_with_connect_info::<SocketAddr>()` 注入对端 IP
- 握手 handler 用 `ConnectInfo<SocketAddr>` 提取真实 IP，拼 `<remote_ip>:<req.listen_port>`，避免对端不知道自己 LAN IP

### 5.8 HTTP 客户端
- `reqwest::Client::builder().no_proxy()` 强制绕过系统代理（Clash/VPN）
- 默认超时 5s，握手 35s（给对方 30s 审批）
- TLS 依赖 `default-features`（Windows schannel / macOS SecureTransport），避免 rustls crypto provider 坑

## 10. 重建提示（给 Claude 或后继开发者）

若要据此文档复刻工程，推荐顺序：

1. `npm create tauri-app@latest sync-copy -- --template svelte-ts` 起骨架
2. 按第 7 节目录结构建各 Rust 模块的空壳
3. 按第 4.2 节定义 `AppState` 和所有辅助 value 类型
4. 按第 5 节定义所有 protocol DTO（`serde::{Serialize, Deserialize}`）
5. 按第 5 节顺序实现各端点 handler（先握手 + 剪切板走通 2 台机，再补其它）
6. `clipboard.rs` 用独立 std::thread + arboard，channel 命令模式
7. `crypto.rs` 封 X25519 + HKDF + AES-GCM
8. `network/client.rs` 的每个 `broadcast_*` 函数都用一样的模板：读配置 → snapshot peers → build reqwest → spawn per-peer tasks → await all（如果调用方需要等完）或直接 fire-and-forget
9. `commands.rs` 把所有前端需要的操作暴露为 `#[tauri::command]`，注意 Tauri 2 的权限系统需要在 `capabilities/default.json` 里显式允许 window API
10. 前端：Svelte 5 runes + `@tauri-apps/api/core` 的 `invoke` + `listen`；一个页面搞完；CSS 用 :global 做 box-sizing 重置
11. CI 用 `tauri-apps/tauri-action` 或手动 `npm run tauri build` + upload-artifact

核心原则：
- **加密只是隐私手段，审批才是身份手段**
- 所有状态 reactive → 前端靠 event 触发 refresh
- 所有涉及用户决定的等待（审批）都用 `oneshot::channel<bool>` + timeout 封装
- 所有广播都是 fire-and-forget，但给上游机会用 `tokio::time::timeout` 限总时长
- 所有接收端都先校验 peer 是否已认证（防止 LAN 外人乱发）

## 11. 已知限制

1. **只有文本、图片、文件三种内容**。富文本（HTML）、代码块、表格等只取纯文本表示
2. **文件 5 MB 上限**。大文件需要分片传输，暂不支持
3. **无持久化信任**：app 重启后 approved_device_ids 清空；要重新走一次审批
4. **心跳 10s 间隔**：peer 突然断网最多 20 秒后才被剔除
5. **Linux 未验证**：arboard 在 Wayland 下对图片剪切板支持不稳定；`macos-private-api` 功能对 Linux 无意义
6. **无端到端文件流式传输**：5 MB 文件整体加载入内存再加密发送，发送期间有短暂的 RAM 峰值
7. **ConnectInfo 只拿到 IP，端口用请求体声明**：对端如果在 NAT 后（跨路由器）无法被正确回连——这是 LAN 限定工具的固有局限

## 12. 版本演进要点（给回溯者参考）

| 阶段 | 关键决策 |
|---|---|
| M1 | Tauri 2 + SvelteKit 脚手架 + 浮窗 + 托盘 |
| M2 | arboard 剪切板轮询 + 前端历史列表 |
| M3 | axum 服务端 + reqwest 客户端 + 握手/广播 + 明文密码认证 |
| M4（早期） | 计划 PBKDF2 + AES-GCM；后因去密码化废弃 |
| M4（最终） | **X25519 ECDH + HKDF + AES-256-GCM**；密钥临时、只存内存、握手时协商 |
| M5 | Gossip mesh（peers 列表在握手响应里同步）+ trust/ban 传播 |
| M6 | 审批弹框 + 文件拖传 + 图片剪切板 + 同步删除 |
| M7 | 分布式审批（所有 peer 同时弹框，任一响应即生效）+ 心跳检测 + leave 广播 |
| UX 迭代 | 边缘吸附 → 最终去除；悬浮球收缩；底部 IP:PORT；小组 N 台计数含自己 |

最终形态：一个用户无需填密码、一人负责审批、端到端加密、支持文本/图片/文件的轻量 LAN 同步工具。
